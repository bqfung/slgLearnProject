# 技术难点学习文档

本文档用于记录 SLG 学习项目中的技术难点、实现思路、常见坑点和面试可讲内容。

## 0. 项目方向调整

### 当前方向

项目方向明确为 SLG 核心系统 Demo。

核心循环：

1. 基地产出资源。
2. 消耗资源升级建筑。
3. 建筑解锁或强化资源、部队、科技。
4. 训练士兵并组成编队。
5. 选择地图据点出征。
6. 战斗结算奖励、损耗和进度。
7. 回到基地继续成长。

### SLG 核心学习点

- 资源系统：产出、消耗、容量、离线收益。
- 建筑系统：建造、升级、效果、解锁关系。
- 科技系统：前置条件、研究消耗、全局加成。
- 部队系统：兵种配置、训练、库存、编队。
- 战斗结算：战力计算、兵种克制、损耗和奖励。
- 地图推进：据点解锁、章节进度、奖励配置。
- 存档系统：版本号、数据迁移、离线收益计算。

### 已有系统复用

- `ComponentPool<T>`、`BulletPool`、`DamageNumberPool`、`HitEffectPool` 可以继续用于后续战斗表现。
- `EnemyHealth`、`EnemyPool` 可以继续用于出征战斗或测试战斗场景。
- `SquadManager` 的人数和阵型思路可以复用到部队展示或战斗表现。
- `VisualConfig` 可以继续作为临时表现配置入口。
- `LevelConfig` 和编辑器校验工具暂时保留，后续可以迁移为 SLG 配置校验工具。

### 后续优先级

1. 基地与资源最小闭环。
2. 建筑升级系统。
3. 部队训练与编队。
4. 地图据点与战斗结算。
5. 科技树与全局加成。
6. 存档、离线收益和配置校验。

## 13. SLG 基地与资源系统

### 目标

实现 SLG 最小局外循环：资源随时间产出，玩家消耗资源升级建筑，建筑升级后反过来提升成长效率。

### 当前实现记录

- `ResourceConfig` 定义资源 id、显示名、初始数量和容量。
- `BuildingConfig` 定义建筑 id、显示名和各等级配置。
- `BuildingLevelConfig` 定义升级耗时、升级消耗和当前等级生产。
- `ResourceInventory` 管理资源数量、容量限制、增加和消耗。
- `BuildingRuntimeState` 管理单个建筑等级、升级倒计时和满级状态。
- `SlgBaseRuntime` 负责初始化资源和建筑，按 `Time.deltaTime` 产出资源，并处理建筑升级。
- `RuntimeBaseUiInstaller` 在 Play 后创建基地 UI，避免编辑器生成场景时丢失运行时引用。
- `BaseHudController` 展示资源、建筑等级、生产、升级消耗和升级按钮状态。
- `StageEightSceneBuilder` 可以一键生成第 8 阶段测试场景和默认配置资产。
- `SlgSaveData` 保存资源数量、建筑等级、升级剩余时间和保存时间。
- `SlgSaveSystem` 使用 `PlayerPrefs + JsonUtility` 保存和读取第 8 阶段 Demo 存档。
- `SlgBaseRuntime` 读取存档后会计算离线时间，并按建筑产出结算离线资源。
- `SLG Learn > Clear Stage 08 SLG Save` 可清空第 8 阶段测试存档。

### 设计取舍

- 当前资源使用字符串 id 关联，方便早期快速创建资产；后续如果配置规模变大，可以改成资源引用或统一 id 校验工具。
- 建筑升级消耗使用“下一等级配置”的成本，含义是从当前等级升级到下一等级。
- 当前建筑生产直接由当前等级配置提供，适合先验证资源循环。
- UI 暂时运行时创建，目的是减少 Prefab 依赖，优先验证系统关系。
- 当前存档使用 `PlayerPrefs`，适合学习 Demo；正式项目更常用文件存档、账号服务器或云端存档。
- 当前离线收益是简化版本：先按保存时建筑等级结算资源，再推进升级倒计时；后续可以做更精确的分段收益。

### 面试可讲点

- SLG 的核心不是单个战斗，而是资源如何进入、消耗、转化为成长，再推动下一次目标。
- 配置数据和运行时状态要分离：建筑配置描述每级规则，建筑运行时状态保存当前等级和升级剩余时间。
- 资源系统要统一处理容量和消耗校验，避免每个 UI 或功能各自扣资源。
- UI 不直接改资源数字，而是调用运行时系统接口，例如 `TryUpgrade`。
- 早期用运行时 UI 验证逻辑，后续可以平滑迁移为正式 UI Prefab。
- 资源每帧增长时不能每次都立刻写存档，应该标记 dirty，再定时保存或在退出时保存。
- `JsonUtility` 不适合直接序列化字典，所以存档中使用 `List<ResourceSaveData>` 和 `List<BuildingSaveData>`。

## 14. 部队训练系统

### 目标

让资源可以转化为部队库存和战斗力，形成 SLG 中“生产资源 -> 训练士兵 -> 准备出征”的第二个核心循环。

### 当前实现记录

- `TroopConfig` 定义兵种 id、显示名、单兵战力、训练耗时和训练成本。
- `TroopInventory` 管理各兵种库存，并提供总战力计算。
- `MarchFormation` 管理当前出征编队，并提供编队战力计算。
- `TroopTrainingOrder` 表示一条训练订单，保存目标兵种和剩余训练时间。
- `SlgBaseRuntime.TryTrain` 负责检查资源、扣除训练成本，并创建训练订单。
- `SlgBaseRuntime.TryAddToFormation` 会从库存扣除 1 个士兵并加入编队。
- `SlgBaseRuntime.TryRemoveFromFormation` 会从编队移出 1 个士兵并返还库存。
- `SlgBaseRuntime.TickTraining` 推进训练倒计时，完成后向 `TroopInventory` 增加士兵。
- 存档新增 `TroopSaveData` 和 `TrainingOrderSaveData`，可以保存兵种库存、编队数量和未完成训练订单。
- `BaseHudController` 和 `RuntimeBaseUiBuilder` 增加部队面板，展示库存数量、编队数量、单兵战力、训练成本、库存战力和编队战力。
- `StageEightSceneBuilder` 默认创建 `Infantry` 和 `Archer` 两种兵种配置。

### 设计取舍

- 当前一次点击训练 1 个士兵，便于先验证资源消耗、倒计时和库存增长。
- 当前允许多个训练订单同时存在，但 UI 只按兵种统计正在训练的数量。
- 当前库存战力和编队战力都是 `兵种数量 * 单兵 Power` 的简单求和，后续可接入兵种克制、科技加成和编队限制。
- 当前编队每次只加入/移出 1 个士兵，避免一开始就引入数量输入和复杂交互。
- 训练订单纳入存档和离线推进，保证退出游戏后训练不会丢失。
- 编队纳入存档，保证退出游戏后当前出征准备不会丢失。

### 面试可讲点

- SLG 的资源系统要能流向多个成长目标，例如建筑升级和部队训练。
- 训练订单是典型的异步成长任务，需要保存剩余时间，并在离线时继续推进。
- 兵种配置、库存和训练订单要分离：配置是静态规则，库存是玩家资产，订单是进行中的任务。
- 库存和编队要分离：库存代表基地资产，编队代表当前出征选择，后续战斗损耗应该优先作用在编队上。
- 战力先用简单公式实现闭环，再逐步加入科技、建筑、兵种克制和编队加成。

## 15. 地图据点与战斗结算

### 目标

实现 SLG 的出征闭环：玩家把编队派去据点，系统根据战力公式结算胜负，发放奖励并处理士兵损耗。

### 当前实现记录

- `StrongholdConfig` 定义据点 id、显示名、推荐战力、胜利损耗率、失败损耗率、驻军、首通奖励、重复奖励、前置据点 id 列表和地图节点坐标。
- `StrongholdGarrisonUnit` 定义据点驻军中的兵种 id 和数量。
- `MarchOrder` 定义一次行军订单，保存目标据点、出征兵种数量和剩余行军时间。
- `BattleReport` 保存最近一次战斗的据点、编队战力、敌军战力、胜负、损耗数量和奖励类型。
- `SlgBaseRuntime.TryAttackStronghold` 负责创建行军订单，并清空当前编队。
- `SlgBaseRuntime.TickMarches` 推进行军倒计时，到达后调用战斗结算。
- `SlgBaseRuntime.IsStrongholdUnlocked` 会优先读取据点的显式前置条件。
- `SlgBaseRuntime.CalculateStrongholdEnemyPower` 根据据点驻军计算敌军战力。
- 胜负公式：`MarchPower >= EnemyPower`。
- 如果据点没有驻军配置，则敌军战力回退为 `RecommendedPower`，兼容旧配置。
- 首次胜利时发放 `StrongholdConfig.FirstClearRewards`，并把据点记录为已攻克。
- 重复挑战已攻克据点胜利时发放 `StrongholdConfig.RepeatRewards`，不会再次发放首通奖励。
- 战斗后会按损耗率从行军订单中的出征士兵里扣除损失。
- 未损失的士兵会回到 `TroopInventory`。
- 点击出征后编队立即清空，`March Power` 归零，士兵进入行军订单。
- 胜利后会把据点 id 记录到已攻克集合。
- 如果据点配置了前置据点 id，则所有前置据点都攻克后才会解锁。
- 如果据点没有配置前置条件，则保留旧的数组顺序解锁逻辑，兼容早期配置。
- 已攻克据点可以重复挑战，但奖励切换为重复奖励，避免重复刷首通奖励。
- `SlgSaveSystem` 会保存已攻克据点 id 和行军订单，重进游戏后恢复地图推进和行军状态。
- 离线分段结算会把行军订单作为时间型事件推进，离线到达后会自动结算战斗。
- `RuntimeBaseUiBuilder` 在左侧生成轻量章节地图节点，并根据前置据点关系绘制分支连线。
- `BaseHudController` 展示据点敌军战力、驻军构成、首通/重复奖励、前置条件、Locked/Unlocked/Cleared 状态、Attack 按钮和最近一次战报。
- `WorldSlgVisualBuilder` 会生成世界中的基地建筑、训练点、地图据点、连线和 3D 操作牌，默认运行场景已从屏幕调试 UI 切换为 GameObject 可视化表现。
- 地图节点和连线会根据状态变色：锁定、可攻打、已攻克分别使用不同颜色，便于玩家理解推进状态。
- `StageEightSceneBuilder` 默认创建 `Outpost`、`Raider Camp` 和 `Supply Depot`，其中后两个据点都依赖 `Outpost`。
- `StageEightSceneBuilder` 会为默认据点写入地图节点坐标，让 `Outpost` 分叉连接到两个后续目标。
- 配置校验会检查据点首通奖励和重复奖励的资源 id 是否有效。
- 配置校验会检查据点前置 id 是否为空、是否引用不存在的据点、是否引用自己、是否重复。
- 配置校验会检查据点驻军兵种 id 是否存在、数量是否大于 0，并对空驻军给出 Warning。

### 设计取舍

- 先做战报公式，不做战斗过程，目的是优先验证 SLG 资源、部队、奖励和损耗流转。
- 敌军战力由驻军构成推导，而不是只依赖一个战力数字，后续更容易接入兵种克制、侦查和战斗模拟。
- `RecommendedPower` 暂时保留为旧配置兜底，也可作为策划参考值或 UI 推荐值。
- 据点解锁现在采用“显式前置条件优先，数组顺序兜底”的方式，既能支持分支地图，也能兼容早期线性配置。
- 节点坐标放在据点配置中，前置关系只负责逻辑解锁和连线生成，两者分离后更容易调整地图表现。
- 当前地图 UI 仍用运行时临时 UI 生成，适合学习阶段快速验证；后续可以替换成正式 Prefab，同时保留配置和运行时逻辑。
- 当前世界表现仍用运行时 Primitive 和 TextMesh 生成，目的是先让玩法“像游戏一样可看可点”；后续可以逐步替换成正式 Prefab、美术模型和动画。
- 首通奖励和重复奖励拆开：首通奖励服务地图推进和成长爆点，重复奖励服务日常刷资源或后续扫荡入口。
- 行军订单和当前编队分离：编队是出征前的选择，行军订单是已经离开基地的资产快照。
- 损耗率直接配置在据点上，后续可以改为战斗公式配置或受科技/兵种影响。
- 胜负公式故意简单，后续再加入兵种克制、随机浮动、科技加成和英雄加成。

### 面试可讲点

- SLG 战斗可以先用公式战报实现主循环，不必一开始做完整战斗表现。
- 出征编队和基地库存分离后，战斗损耗可以自然作用在编队上，剩余士兵再回到库存。
- 加入行军后，战斗损耗应作用在行军订单上，而不是当前编队或基地库存，避免玩家在等待期间改编队导致结算错乱。
- 奖励发放应该走统一资源系统，避免战斗结算直接改 UI。
- 一次性奖励和重复奖励必须分开建模，否则重复挑战会破坏资源经济。
- 地图进度应该存储稳定 id，而不是存数组下标，避免配置重排导致玩家进度错乱。
- 地图解锁应该配置化并被校验，避免内容配置改动后出现死路、未知引用或自引用。
- 地图 UI 的表现状态应从同一套运行时状态计算，避免“按钮可点、节点还显示锁定”这类展示不一致。
- 表现层可以重做，但底层资源、建筑、部队、行军和地图结算逻辑不应跟着重写；这是分层架构的价值。
- 战斗公式要保留从简单到复杂的演进路径：推荐战力兜底 -> 驻军战力 -> 兵种克制 -> 随机浮动/科技/英雄加成。
- 战斗报告是 UI 展示层和结算逻辑之间的轻量数据边界。

## 16. 科技树与全局加成

### 目标

实现 SLG 的长期成长系统：玩家消耗资源研究科技，科技完成后反过来影响资源、训练和战斗数值。

### 当前实现记录

- `TechnologyConfig` 定义科技 id、显示名和等级列表。
- `TechnologyLevelConfig` 定义研究耗时、研究消耗、科技前置、建筑等级条件和科技效果。
- `TechnologyPrerequisite` 定义需要先达到的科技等级。
- `BuildingLevelRequirement` 定义需要先达到的建筑等级。
- `TechnologyEffect` 定义效果类型、目标 id 和加成值。
- `TechnologyRuntimeState` 保存单个科技的等级和研究剩余时间。
- `SlgBaseRuntime.TryResearch` 负责检查前置条件、资源、扣除研究成本并开始研究倒计时。
- `SlgBaseRuntime.TickTechnologies` 推进研究倒计时，完成后提升科技等级。
- `SlgBaseRuntime.AreTechnologyRequirementsMet` 是科技前置条件的统一判断入口。
- `SlgSaveData` 和 `SlgSaveSystem` 保存科技等级和研究剩余时间。
- `RuntimeBaseUiBuilder` 在右侧生成科技列表。
- `BaseHudController` 展示科技等级、研究消耗、倒计时、效果和 Research 按钮。
- 示例科技：
  - `Agriculture`：提升 Food 产出。
  - `Military Drill`：提升训练速度，并在高等级提升部队战力。

### 当前效果接入点

- 资源产出：建筑产出时乘以 `ResourceProductionMultiplier`。
- 训练时间：创建训练订单时用训练速度加成缩短剩余时间。
- 部队战力：库存战力、编队战力、据点战斗结算都使用加成后的单兵战力。

### 设计取舍

- 当前科技配置已经支持前置科技和建筑等级条件，但 UI 仍是列表视图，还不是正式树状视图。
- 科技效果使用 `type + targetId + value`，比为每个科技写一段逻辑更容易扩展。
- `targetId` 为空表示影响全部目标，例如所有兵种训练速度或所有兵种战力。
- 加成值使用小数配置，例如 `0.2` 表示 `+20%`。
- 当前把科技类放在现有脚本文件中，便于当前工程立即编译；后续 Unity 重新生成项目文件后，可以拆分成独立文件。

### 面试可讲点

- 科技系统本质是“长期成长”和“全局修正器”，不要让 UI 直接改变资源或战力。
- 配置定义效果，运行时统一读取效果并作用到产出、训练和战斗公式，能减少硬编码。
- 科技前置条件应在运行时和 UI 共用同一套判断，避免按钮显示可点但逻辑拒绝，或反过来。
- 科技进度要存等级和剩余时间，离线时也要推进研究倒计时。
- 数值加成要在统一计算入口生效，例如 `CalculateFormationPower`，避免 UI 和战斗显示不一致。

## 17. 存档容错、离线收益与配置校验

### 目标

让 SLG Demo 的长期成长更稳定：功能持续增加后，旧存档不应轻易读崩，配置错误也应尽量在编辑器阶段被发现。

### 当前实现记录

- `SlgSaveData.CurrentVersion` 记录当前存档结构版本。
- `SlgSaveData.Normalize` 会补齐可能为空的列表字段。
- `SlgSaveMigrator.MigrateToCurrent` 是统一存档迁移入口。
- 当前存档版本为 3，新增 `marchOrders` 用于保存行军订单。
- `SlgSaveSystem.TryLoad` 读取 JSON 后统一调用迁移入口。
- `SlgSaveSystem.Save` 写入当前版本号。
- `SlgBaseRuntime.ApplyOfflineProgress` 现在会分段推进资源、建筑、训练、行军和科技研究。
- `ApplyOfflineProgressSegments` 会找到最近的时间型事件，并在事件完成后用新状态继续结算剩余离线时间。
- `SLG Learn > Validate All SLG Configs` 会扫描 `Assets/_Project/ScriptableObjects` 下的 SLG 配置资产。
- `SLG Learn > Config Validator` 提供独立编辑器窗口，可以扫描、统计并查看所有校验问题。
- 校验窗口中的 `Select` 按钮会选中并高亮对应资产。
- `SlgConfigBuildPreprocessor` 会在 Unity Build 前自动执行同一套配置校验。
- 构建前校验遇到 Error 会抛出 `BuildFailedException` 阻止构建，Warning 只输出日志。
- 校验结果会带上具体资产路径，便于从 Console 追到问题配置。
- 当前配置校验覆盖：
  - 资源 id、容量和初始数量。
  - 建筑等级、升级消耗和产出资源引用。
  - 兵种 id、战力、训练时间和训练消耗。
  - 据点 id、推荐战力、奖励资源和推进难度顺序。
  - 科技 id、研究消耗、研究时间、前置科技、建筑条件、效果目标和目标 id。

### 设计取舍

- 当前存档迁移先做“缺字段容错”和版本入口，因为项目还处于学习 Demo 阶段，复杂字段改名或拆分迁移可以等字段稳定后再加。
- 当前配置校验同时提供菜单入口、Console 输出、基础窗口和构建前自动阻断，能快速保护项目中的 ScriptableObject 配置。
- 当前校验工具按资产类型扫描 `ScriptableObjects` 目录，后续可以升级为带搜索、过滤和字段定位的完整窗口。
- 离线结算已按建筑升级、训练完成和科技完成进行分段，但还没有生成可展示给玩家的离线收益报告。

### 面试可讲点

- 存档字段会随版本增加，新增列表字段必须考虑旧存档反序列化后的空引用。
- 存档迁移要集中在统一入口中，避免各个系统读取时各自做兼容逻辑。
- 配置校验应尽量检查“引用是否存在”和“数值是否合理”，避免运行时才发现资源 id 写错。
- 校验规则应抽成可复用方法，让菜单、窗口、构建前检查共用同一套规则。
- 构建前校验可以作为质量门禁，避免错误配置进入包体或交付版本。
- 科技前置条件要校验引用是否存在、是否自引用、要求等级是否超过最大等级。
- 离线收益不是简单加资源，还要推进建筑、训练、科技等所有时间型任务。
- 离线分段结算能避免“建筑中途升级完成，但全程按旧等级产出”的数值误差。
- 对学习项目来说，先做最小迁移和校验闭环，再逐步演进成完整工具，是更稳的开发路径。

## 1. 小队移动系统

### 目标

实现玩家小队自动向前移动，并允许玩家通过左右拖拽控制横向位置。

### 技术点

- 输入系统：鼠标、触摸、键盘调试输入。
- 移动方式：Transform、CharacterController、Rigidbody 的取舍。
- 边界限制：限制玩家横向移动范围。
- 摄像机跟随：LateUpdate 中跟随目标，减少抖动。

### 难点

- 移动端触摸输入和编辑器鼠标输入兼容。
- 前进速度与横向速度手感调节。
- 横向移动不能穿出赛道。
- 摄像机跟随不能明显抖动。

### 面试可讲点

- 为什么角色移动常放在 Update，而物理相关移动常放在 FixedUpdate。
- LateUpdate 适合处理摄像机跟随的原因。
- Transform 移动和 Rigidbody 移动的区别。

### 当前实现记录

- `SquadController` 负责小队整体移动。
- 小队每帧自动向世界 Z 轴正方向前进。
- 横向输入同时支持键盘 A/D、方向键、鼠标拖拽和触摸拖拽。
- 横向位置通过 `horizontalLimit` 限制，避免移出赛道。
- 当前使用 Transform 位移，原因是第 1 阶段重点是跑酷控制手感，不涉及复杂物理碰撞。
- `CameraFollow` 在 LateUpdate 中跟随小队，减少摄像机和玩家移动的时序抖动。
- `StageOneSceneBuilder` 是编辑器工具，用于自动生成测试场景，避免每次手动创建对象和挂脚本。

## 2. 小队阵型系统

### 目标

根据当前人数动态生成和排列士兵。

### 技术点

- 阵型坐标计算。
- 士兵对象池。
- 人数增加和减少。
- 阵型刷新动画。

### 难点

- 人数频繁变化时避免反复 Instantiate 和 Destroy。
- 阵型要既整齐又不拥挤。
- 减员时选择哪些士兵移除。

### 面试可讲点

- 对象池如何减少 GC 和实例化开销。
- 数据结构如何管理当前存活士兵。
- 阵型刷新如何避免瞬间跳变。

### 当前实现记录

- `SquadManager` 管理当前小队成员列表。
- `AddMembers`、`RemoveMembers`、`SetCount` 提供统一人数变化接口，后续倍率门可以直接调用。
- 当前阵型按固定列数排列，行内居中，人数变化后重新计算目标位置。
- 士兵位置使用 Lerp 平滑移动到目标位置，避免人数变化时瞬间跳变。
- 如果没有配置士兵 Prefab，会自动创建胶囊体作为临时士兵，方便早期测试。
- `SquadDebugInput` 用于开发期测试，按 E 增加人数，按 Q 减少人数。

## 3. 倍率门系统

### 目标

玩家通过门时，小队人数根据门的规则变化。

### 技术点

- Trigger 碰撞检测。
- 门数据配置。
- UI 文本显示。
- 触发后禁用重复触发。

### 难点

- 避免一个门被多个士兵重复触发。
- 加法、减法、乘法等规则统一抽象。
- 负数或人数为 0 的边界处理。

### 面试可讲点

- Collider 和 Trigger 的区别。
- 如何设计可扩展的门规则。
- 为什么要把门数值做成配置。

### 当前实现记录

- `GateOperation` 定义门的运算类型：加法、减法、乘法。
- `SquadGate` 负责检测小队通过，并调用 `SquadManager` 的人数接口。
- 门内部用 `hasTriggered` 防止多个士兵重复触发同一个门。
- 门触发后会关闭自身渲染和碰撞，作为临时反馈。
- `StageTwoSceneBuilder` 是编辑器工具，用于自动生成包含多组倍率门的测试场景。

### 当前设计取舍

- 门不直接管理士兵对象，只调用 `SquadManager`，这样倍率门逻辑和阵型管理保持解耦。
- 加、减、乘统一通过枚举表达，后续可以扩展为除法、固定人数、随机奖励等规则。
- 当前数值写在场景工具中，后续第 6 阶段会迁移到 ScriptableObject 或关卡配置中。

## 4. 自动索敌与攻击

### 目标

小兵自动寻找合适目标并按攻击频率输出伤害。

### 技术点

- 目标搜索范围。
- 攻击冷却。
- 射线、子弹、范围检测。
- 目标优先级。

### 难点

- 多个小兵频繁搜索敌人可能导致性能压力。
- 敌人死亡后目标引用失效。
- 攻击频率、射程、伤害要可配置。

### 面试可讲点

- 如何优化大量单位的索敌逻辑。
- Update 中做搜索为什么可能有性能问题。
- 事件驱动和轮询的区别。

### 当前实现记录

- `SoldierWeapon` 挂在每个士兵身上，按攻击间隔寻找目标并发射子弹。
- 目标搜索使用 `Physics.OverlapSphereNonAlloc`，避免每次搜索都分配新数组。
- 当前优先攻击前方攻击范围内距离最近的敌人。
- `EnemyHealth` 负责敌人的血量、受击和死亡。
- 普通敌人死亡后通过 `EnemyPool` 回收，Boss 死亡后用于触发胜利判断。

### 当前设计取舍

- 当前攻击已经升级为对象池子弹，命中后再造成伤害。
- 每个士兵独立索敌，逻辑清晰但规模变大后会有性能压力；后续可以改成小队级目标分发、分帧搜索或空间分区。
- 当前只检查目标是否在前方，暂未处理遮挡、目标权重和多目标技能。

## 5. 子弹与对象池

### 目标

通过对象池复用子弹、伤害数字和特效。

### 技术点

- 泛型对象池。
- 预加载。
- 取出、回收、重置状态。
- 生命周期管理。

### 难点

- 回收对象时必须重置状态。
- 子弹命中、超时、场景切换时都要正确回收。
- 池容量过小或过大都会有问题。

### 面试可讲点

- Instantiate 和 Destroy 的性能成本。
- 对象池如何减少 GC。
- 对象池可能带来的状态残留问题。

### 当前实现记录

- `BulletPool` 是运行时子弹对象池，首次攻击时会自动创建。
- `BulletPool` 会预热一批子弹，并在发射时从队列中取出。
- `Bullet` 负责追踪目标、移动、命中、超时和回收。
- `SoldierWeapon` 不再直接对敌人扣血，而是在攻击冷却结束后向目标发射子弹。
- 子弹命中敌人后调用 `EnemyHealth.TakeDamage`，然后回收到池中。
- `DamageNumberPool` 负责复用伤害数字。
- `HitEffectPool` 负责复用命中闪光效果。
- 子弹命中后会同时触发伤害数字和命中特效，二者各自到期后回收。
- `EnemyPool` 负责复用普通敌人。
- `PooledEnemy` 监听 `EnemyHealth.Died` 事件，并在普通敌人死亡时通知对象池回收。
- `EnemySpawner` 现在通过 `EnemyPool` 生成普通敌人，Boss 仍独立创建。
- `ComponentPool<T>` 抽出了对象池的通用逻辑：预热、取出、回收、超过配置容量时警告。
- `BulletPool`、`DamageNumberPool`、`HitEffectPool`、`EnemyPool` 都继承 `ComponentPool<T>`。
- `ComponentPool<T>` 暴露 `TotalCreated`、`AvailableCount`、`ActiveCount`、`PeakActiveCount`、`ExpansionCount`，供调试面板读取。
- `PoolDebugOverlay` 会显示各对象池的 active/free/total/peak/grow 数量，用于运行时观察池复用情况。
- `PoolDebugOverlay` 支持按 F3 隐藏或显示。

### 当前设计取舍

- 对象池当前覆盖子弹、伤害数字和命中特效，仍保持在战斗反馈范围内。
- 普通敌人也已经接入对象池，死亡后不再只做隐藏，而是回到 `EnemyPool`。
- 子弹和命中特效使用运行时球体，伤害数字使用 TextMesh，后续都可以替换为正式 Prefab。
- `BulletPool` 当前是简化版全局池，适合 Demo；大型项目中可以按场景、玩法模块或资源类型管理多个池。
- Boss 暂不池化，因为它通常是关卡关键对象，生命周期与普通敌人波次不同。
- 通用池不关心具体对象怎么创建和怎么播放，只管理池生命周期，避免抽象层过度了解业务。

### 面试可讲点补充

- 对象池不是为了让单个对象更快，而是为了避免频繁 `Instantiate` / `Destroy` 造成 CPU 峰值和 GC 压力。
- 回收对象时必须重置运行时状态，例如目标引用、生命周期、位置、激活状态。
- 对象池容量需要结合峰值并发量设置，太小会频繁扩容，太大则浪费内存。
- 反馈类对象特别适合池化，因为它们数量多、生命周期短、创建销毁频繁。
- 敌人池化时要特别注意状态重置，例如血量、目标、移动速度、攻击冷却、父节点和激活状态。
- 当前 `EnemyMeleeAttacker.Configure` 会重置攻击冷却，避免池化敌人复用后继承旧状态。
- 抽通用池时要避免“为了复用而复用”：如果通用基类塞入太多业务参数，反而会让每个对象池更难维护。
- 对象池调试面板能帮助判断池是否在复用：如果 total 长期上涨，说明池容量不够或回收逻辑有问题；如果 grow 大于 0，说明运行时已经超过了预设池容量。

## 9. 数据驱动设计

### 目标

通过配置管理角色、敌人、武器、关卡数据。

### 技术点

- ScriptableObject。
- 配置引用。
- 编辑器可视化。
- 运行时读取配置。

### 难点

- 配置之间的引用关系要清晰。
- 修改配置后要避免影响运行时临时状态。
- 区分静态配置和动态存档数据。

### 面试可讲点

- ScriptableObject 的优点和注意事项。
- 配置数据和运行时数据为什么要分离。
- 数据驱动如何提升可扩展性。

### 当前实现记录

- `LevelConfig` 是关卡配置入口，包含赛道长度、宽度、倍率门、敌人波次和 Boss 配置。
- `GateConfig` 描述一组左右门的位置、运算类型和数值。
- `EnemyWaveConfig` 描述敌人波次的位置、数量、血量、速度、攻击范围、攻击频率和减员数量。
- `BossConfig` 描述 Boss 位置、血量、速度、攻击范围、攻击频率和减员数量。
- `StageSixSceneBuilder` 首次执行时会自动创建默认 `Stage06_LevelConfig.asset`，并创建挂载 `LevelBuilder` 的测试场景。
- `LevelBuilder` 在运行时读取 `LevelConfig`，生成赛道、小队、摄像机、倍率门、敌人波次、Boss 和 UI。

### 当前设计取舍

- 现在采用 ScriptableObject 作为静态关卡配置，适合 Unity 编辑器内调参与版本管理。
- 配置类只保存数据，不直接创建 GameObject，避免数据层和表现层耦合。
- 当前生成逻辑已经从 Editor 工具迁移到运行时 `LevelBuilder`，Editor 菜单只负责创建配置资产和测试场景。
- `LevelBuilder` 当前职责较多，后续可以按对象类型继续拆分，避免变成新的上帝类。

### 当前拆分记录

- `GateBuilder` 负责根据 `GateConfig` 创建左右倍率门、门面板和文字。
- `EnemySpawner` 负责根据 `EnemyWaveConfig` 创建普通敌人，并根据 `BossConfig` 创建 Boss。
- `RuntimePrimitiveFactory` 暂时集中处理运行时临时材质创建，避免颜色和 Shader 查找逻辑散落在多个类里。
- `RuntimeUiBuilder` 负责创建战斗 HUD、结果面板、重开按钮和 EventSystem。
- `EnvironmentBuilder` 负责运行时创建临时灯光、赛道和边界。
- `PlayerSquadFactory` 负责创建玩家小队对象并挂载小队相关组件。
- `RuntimeCameraFactory` 负责创建主摄像机并挂载跟随组件。
- `LevelBuilder` 现在主要负责关卡构建顺序：环境、小队、摄像机、倍率门、敌人、Boss、胜负控制和 UI 装配。
- `VisualConfig` 负责集中配置运行时临时表现，包括颜色、文字大小和基础尺寸。
- `RuntimePrimitiveFactory` 在 `LevelBuilder` 启动时接收 `VisualConfig`，运行时创建临时对象时统一读取这些表现参数。
- `VisualConfig` 现在也可以配置子弹、命中特效、敌人、Boss、士兵 Prefab。
- `RuntimePrimitiveFactory.InstantiatePrefabOrPrimitive` 负责“优先实例化 Prefab，未配置时回退到临时 Primitive”的逻辑。

### 面试可讲点补充

- 一个类开始变大时，不一定马上做复杂架构；可以先按变化原因拆分，例如门、敌人、UI 的变化原因不同。
- `LevelBuilder` 作为流程编排器，不应长期持有所有对象创建细节，否则会逐渐变成上帝类。
- 拆分后每个类更容易单独替换，例如后续可以把 `EnemySpawner` 改成对象池版本，而不影响倍率门逻辑。
- UI 创建拆到 `RuntimeUiBuilder` 后，后续可以从“代码创建 UI”平滑替换为“加载 UI Prefab”，不用改关卡生成流程。
- 环境、小队和摄像机拆出后，`LevelBuilder` 更接近流程编排器；后续替换正式 Prefab 时影响面更小。
- 表现配置先集中到 `VisualConfig`，再逐步迁移到正式 Prefab，是比一次性替换所有资源更稳的路径。
- Prefab 字段保持可选，可以在没有美术资源时继续运行 Demo，也可以逐步替换单个对象类型。

### 面试可讲点补充

- ScriptableObject 适合保存静态配置，但不适合直接保存玩家运行时存档。
- 配置数据和运行时状态要分离，例如敌人最大血量属于配置，当前血量属于运行时状态。
- 数据驱动的价值是减少硬编码，让策划或开发者可以通过配置快速调整关卡。

## 12. 关卡编辑器

### 目标

通过 Unity Editor 扩展，让 `LevelConfig` 更容易校验、生成和调试，形成作品集亮点。

### 当前实现记录

- `LevelConfigValidator` 负责纯配置校验，不依赖 Inspector。
- `LevelConfigIssue` 用于描述单条校验结果，包含级别、消息和配置路径。
- `LevelValidationSettings` 保存关卡校验阈值，例如 Gate spacing、Enemy wave spacing。
- `LevelConfigEditor` 是 `LevelConfig` 的自定义 Inspector。
- Inspector 中提供 `Validate Config` 按钮。
- Inspector 中提供 `Generate Preview Scene For This Config` 按钮。
- Inspector 中提供 `Validation Settings` 引用，默认使用 `Stage07_LevelValidationSettings.asset`。
- 校验结果通过 `HelpBox` 显示在 Inspector 下方，Error 和 Warning 使用不同样式。
- 菜单 `SLG Learn > Build Stage 06 Data Driven Scene` 仍生成默认测试场景。
- Inspector 生成按钮会基于当前选中的 `LevelConfig` 生成 `Preview_<ConfigName>.unity`。
- 生成预览场景前会自动运行校验；如果存在 Error，会弹窗让开发者选择取消或继续生成。
- 校验结果会显示 `PropertyPath`，例如 `boss.position`、`gates.Array.data[0].z`。
- 校验器会检查关键内容之间的间距，发现节奏过密时给出 Warning。

### 当前校验规则

- Road length 和 road width 必须大于 0。
- Gate Z 不能为负数，不能超过赛道长度。
- Gate 左右数值必须大于 0。
- Enemy wave Z 不能为负数，不能超过赛道长度。
- Enemy wave 应按 Z 从小到大排序。
- Enemy wave count 和 health 必须大于 0。
- Boss 必须存在，血量必须大于 0。
- Boss 应放在最后一波敌人之后。
- VisualConfig 未配置时给出 Warning，因为运行时仍可使用代码回退表现。
- Gate 之间建议间距由 `LevelValidationSettings.MinGateSpacing` 控制，默认 8。
- Enemy wave 之间建议间距由 `LevelValidationSettings.MinWaveSpacing` 控制，默认 10。
- Gate 和 Enemy wave 之间建议间距由 `LevelValidationSettings.MinGateWaveSpacing` 控制，默认 6。
- Boss 与最后一波敌人的建议间距由 `LevelValidationSettings.MinBossAfterLastWaveSpacing` 控制，默认 12。

### 面试可讲点

- Editor 工具可以减少配置错误，提高关卡制作效率。
- 校验逻辑和 Inspector 展示分离，更方便后续复用到批量检查或构建前检查。
- ScriptableObject 适合做关卡配置，自定义 Inspector 适合做配置入口和轻量工具链。
- 工具链是作品集亮点，因为它体现了从“能玩”到“能生产内容”的工程意识。
- 让预览场景基于当前配置生成，比固定生成默认关卡更贴近真实关卡生产流程。
- 生成前校验是常见工具链设计：既要减少低级配置错误，也要允许开发者在需要时带着提醒继续调试。
- 校验结果分级很重要：Error 表示可能破坏关卡运行或数值逻辑，Warning 表示可运行但建议修正。
- `PropertyPath` 让校验结果具备可扩展性，后续可以用于点击定位、批量报告或构建前检查日志。
- 关卡节奏问题适合先做成 Warning，因为它更像设计建议；这样工具不会过度阻断调试流程。
- 校验阈值做成 ScriptableObject 后，工具规则可以被配置和复用，而不是散落在代码常量里。

## 18. 世界可视化降噪

### 问题

把 UI 文字直接搬到 3D 世界里，会很快变成不可读的调试面板：TextMesh 会互相遮挡，镜头距离变化后字体比例也不稳定，玩家看不到“建筑、兵营、据点、出征”这些核心对象。

### 当前调整

- `WorldSlgVisualBuilder` 仍然运行时生成临时对象，但默认表达方式改为 GameObject 优先。
- 建筑状态用立方体高度和颜色表达：等级越高越高，升级中偏黄色，满级偏绿色。
- 部队状态用胶囊体表达：蓝色代表预备兵，绿色代表当前编队。
- 据点状态用球体颜色表达：灰色锁定、蓝色可攻打、绿色已攻克。
- 据点敌军强度用红色胶囊体数量表达，避免把驻军、战力和奖励全部写成文字。
- 文字只保留短标签、资源概览、战力概览和最近战报，操作按钮也缩短为单字符命令。
- `WorldClickAction` 增加可交互状态，按钮变灰时会真正禁止点击。
- 材质创建优先使用 `Universal Render Pipeline/Lit`，避免 URP 项目中使用内置管线 `Standard` Shader 导致对象显示为粉红色。
- 新增选中详情面板：点击建筑、兵种训练点或据点后，才显示升级、训练、编队、奖励、前置条件等详细数值。
- 新增选中高亮和详情操作按钮：被选中对象脚下显示黄色标记，详情面板根据当前状态显示 `Upgrade`、`Train`、`Add`、`Remove` 或 `Attack`。
- 新增科技可视化入口：科技节点进入沙盘表现，详情面板显示研究消耗、研究时间、前置条件和效果，并提供 `Research` 操作。
- 新增科技前置连线和下一目标提示：科技锁定原因可以从节点连线看出来，顶部 `Next` 提示用于引导玩家继续推进主循环。
- 新增 `Go` 目标跳转：根据当前 `Next` 目标自动选中对应建筑、兵种、科技或据点。
- 新增 `Objectives` 任务列表：最多显示 3 条当前可处理目标，每条都有独立跳转按钮。

### 设计原则

- 常驻信息越少越好，详细数值应该放到选中详情、悬停面板或正式 HUD 中。
- 3D 世界优先用形状、颜色、数量、尺寸表达状态，文字只做命名和确认。
- 学习阶段可以先用 Primitive 快速验证表现规则，后续再替换为 Prefab 和美术资源。
- 表现层可以反复重做，但不应该影响底层资源、建筑、部队、行军和战斗结算逻辑。
- Unity 中粉红色材质通常代表 Shader 缺失或当前渲染管线不支持；URP 项目要优先使用 URP Shader。
- “概览”和“详情”要分层：概览解决玩家看地图，详情解决玩家做决策。
- 操作按钮的启用状态必须来自运行时规则，而不是只看 UI 表面状态；否则会出现按钮可点但逻辑失败，或按钮禁用但其实可执行的问题。
- 科技是 SLG 长线成长的一部分，不应该只藏在后台数值里；哪怕早期用临时节点，也要让玩家看见它如何影响产出、训练和战力。
- 引导目标应从运行时状态推导，而不是手写固定文案；这样资源、训练、编队、行军和章节推进变化后，提示能自动跟随。
- 目标跳转不应该直接替玩家执行操作，只负责定位对象；真正的升级、训练、研究和攻击仍由详情按钮确认。
- 任务列表应该优先覆盖主循环动作，而不是把所有可做事项都列出来；过多目标会稀释玩家决策。

### 面试可讲点

- 玩法原型不是把所有数据都显示出来，而是把玩家需要做决策的信息分层展示。
- 世界表现和系统逻辑要解耦：`WorldSlgVisualBuilder` 只读取运行时状态并调用运行时接口，不直接改资源或战斗结果。
- 图形化状态比大段文字更适合移动端和策略沙盘，后续可以自然演进为正式 Prefab、选中态、详情面板和动效。
- 选中详情面板是典型的表现层适配：同一套底层数据既可以驱动 3D 沙盘，也可以驱动 HUD 信息，不需要复制一套玩法逻辑。
- 选中高亮解决“当前我在操作谁”的反馈问题，详情按钮解决“我下一步能做什么”的决策问题，这是策略游戏交互闭环的一部分。
- 科技节点状态颜色体现了配置条件、资源消耗和异步研究任务三类状态，适合面试时说明“配置数据、运行时状态、表现层”如何协作。
- 科技前置连线体现了数据驱动 UI：不是在场景里手工摆线，而是从 `TechnologyPrerequisites` 自动生成表现。
- 下一目标提示可以作为早期任务系统雏形，后续能演进成任务追踪、自动选中目标和新手引导。
- `Go` 跳转是任务系统的最小交互闭环：运行时状态推导目标，表现层显示目标，点击后定位到可操作对象。
- 多目标任务列表可以展示“主线推进”和“成长补强”两类目标，是 SLG 中任务、章节和新手引导系统的基础形态。

## 19. 兵种差异与克制

### 目标

让部队系统从“数量乘战力”升级为“根据敌军组成选择兵种”，这是 SLG 编队策略的核心之一。

### 当前实现记录

- `TroopConfig` 新增 `Role`、`CounterRole` 和 `CounterMultiplier`。
- 默认兵种扩展为步兵、射手、骑兵和攻城。
- `SlgBaseRuntime.CalculateFormationPowerAgainst` 会根据目标据点驻军主力类型计算针对性战力。
- `SlgBaseRuntime.CalculateMarchOrderPower(order, target)` 会在战斗结算时应用兵种克制倍率。
- `SlgBaseRuntime.CalculateStrongholdBattlePreview` 统一生成战斗预览数据，包括敌军主力、推荐兵种、基础战力、克制后战力和预期结果。
- `SlgBaseRuntime.TryRecommendFormation` 会根据据点主力和库存自动把合适兵种加入编队。
- 世界详情面板会显示兵种类型、克制目标和克制倍率。
- 据点详情中的我方战力显示为针对该据点驻军计算后的战力。
- 任务列表在需要训练时，会优先推荐克制当前章节目标据点的兵种。

### 设计取舍

- 当前先用“据点主力兵种类型”作为克制判断目标，避免一开始就做复杂矩阵和逐兵种对位。
- 克制倍率配置在兵种上，便于不同兵种有不同强度，例如攻城兵克制倍率更高但成本更重。
- 据点敌军战力暂时仍按基础兵种战力计算，克制主要作用在我方出征编队上，便于玩家理解和调试。
- 默认兵种仍然可以被科技按 id 或全体加成影响，克制倍率和科技倍率会共同影响战斗结果。
- 战斗预览只负责提示，不直接改变结算；真正结算仍走 `ResolveMarchOrder`，并复用同一套克制战力计算。
- 推荐编队只改变“出征前编队”，不会自动攻击据点；这样既能降低操作成本，也保留玩家确认权。

### 面试可讲点

- SLG 的战斗策略不应只靠总战力比较，至少要有兵种差异、驻军构成和克制关系。
- 先做简单主力克制，再演进到完整克制矩阵，是学习项目中风险较低的迭代方式。
- 兵种配置、战斗计算和 UI 展示要共用同一套运行时计算入口，避免面板显示能赢但结算失败。
- 克制关系是后续侦查、推荐编队、战斗预览和自动配兵系统的基础。
- 出征前预览是 SLG 中非常关键的体验层：它让玩家理解为什么要训练某类兵、为什么当前编队不够，以及下一步资源应该投向哪里。
- 推荐编队是从“信息提示”走向“辅助决策”的一步，后续可以继续扩展为自动配兵、最小损耗编队和多队列行军。
